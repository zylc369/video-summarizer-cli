# 需求-视频总结-v2

> 本文档可直接作为 AI 开发提示词使用，包含所有必要的技术决策、约束和实现细节。

---

## 1. 需求概述

读取视频讲解内容，通过 AI 总结视频内容，生成结构化的 Markdown 文档。如果视频讲解涉及代码，生成的总结中应包含相关示例代码。

### 1.1 视频来源

- 本地视频文件（MP4、MKV、AVI 等常见格式）
- 局域网 NAS 提供的视频链接（视为本地视频，通过网络路径访问），例如：`http://mydrive.beaconkeep.com:8011/d/%E8%AF%BE%E7%A8%8B/%E5%AE%89%E5%8D%93%E9%80%86%E5%90%91/Android%E9%80%86%E5%90%91%E3%80%90%E6%8C%81%E7%BB%AD%E6%9B%B4%E6%96%B0%E3%80%91/%E5%B0%8F%E8%82%A9%E8%86%80%E9%80%86%E5%90%91%E6%95%99%E7%A8%8B%E8%B5%84%E6%96%99/9.%E3%80%90%E5%B0%8F%E8%82%A9%E8%86%80%E3%80%91APP%E9%80%86%E5%90%91%E7%99%BE%E9%9B%86/%E8%A7%86%E9%A2%91/01%20Android%E4%BD%93%E7%B3%BB%E7%BB%93%E6%9E%84.mp4?sign=YG63FmI5FsuVOmxIyPdFh6xyleBlZjntXEO8xKcWHrA=:0`。

### 1.2 视频语言特征

- 主体为中文讲解
- 讲解代码时必然混杂英文（变量名、函数名、技术术语如 API、React、Docker 等）
- 属于中英混合（code-switching）场景，语言切换频繁

### 1.3 视频视觉内容特征

视频包含以下类型的屏幕画面：

| 画面类型 | 复杂度 | 示例 |
|---------|--------|------|
| IDE 代码编辑器 | 低-中 | VSCode、IntelliJ 等展示源代码 |
| 终端/命令行 | 低 | 滚动的日志、命令执行输出 |
| PPT 演示 | 中 | 文字、图表、代码片段 |
| IDA Pro 反汇编 | 高 | 函数调用关系图、交叉引用、内存布局、汇编代码 |
| 滚动日志 | 中-高 | 快速翻滚的调试日志 |

### 1.4 运行环境

| 设备 | CPU/GPU | 内存/显存 | 角色 |
|------|---------|----------|------|
| Macbook Pro M4 | Apple M4 芯片 | 48GB 统一内存 | 开发机 + 备用执行机 |
| Windows PC | RTX 4070Ti (12GB VRAM) | 128GB RAM | 主要 GPU 执行机 |

两台机器均可用，系统需自动检测硬件环境并选择最优的执行策略。

### 1.5 API 资源

- 智谱 Max Coding Plan，包含大量文本模型配额：GLM-5.1、GLM-5、GLM-5-Turbo、GLM-4.7、GLM-4.6、GLM-4.5-Air
- 智谱视觉模型（按需付费，成本极低）：GLM-5V-Turbo、GLM-4.6V、GLM-4.6V-FlashX、GLM-4.6V-Flash（免费）、GLM-4.5V
- 视频内容无需保密，可以调用远端 API

### 1.6 输出格式

- Markdown 文档
- 支持嵌入关键帧截图（通过相对路径引用提取的图片）
- 支持代码块（从视频中提取或 AI 生成的示例代码）
- 支持结构化标题、列表、表格

## 2. 技术栈决定

**使用 Python。**

理由：
1. ML 生态（faster-whisper、PaddleOCR、PySceneDetect、Qwen3-ASR）全部是 Python 原生
2. 智谱 API 有官方 Python SDK（zhipuai）
3. 不使用前端技术栈（Node.js/Bun），避免生态不匹配导致的封装成本
4. 先实现命令行执行，后续需要 GUI 时再套前端壳（如 Gradio、Streamlit 或 Web UI）

## 3. 整体处理管道

```
视频输入
  │
  ├──→ [1] 音频提取 (FFmpeg)
  │         │
  │         ↓
  │    [2] 语音转文字 (ASR)
  │         │ faster-whisper turbo (本地 GPU)
  │         │ 或 Qwen3-ASR-1.7B (本地 GPU)
  │         │
  │         ↓
  │    转录文本 (带时间戳)
  │
  ├──→ [3] 关键帧提取 (FFmpeg)
  │         │
  │         ↓
  │    [4] 关键帧视觉分析 (智谱视觉 API)
  │         │ GLM-4.6V-FlashX (默认)
  │         │ 可升级至 GLM-4.6V / GLM-5V-Turbo
  │         │
  │         ↓
  │    视觉内容描述 (带时间戳)
  │
  └──→ [5] AI 综合总结 (智谱文本 API)
            │ GLM-4.7 或 GLM-5.1
            │ 输入: 转录文本 + 视觉内容描述
            │ 输出: 结构化 Markdown
            │
            ↓
       最终 Markdown 文档
```

**管道阶段 1-4 可并行执行**：音频提取+ASR 与 关键帧提取+视觉分析 同时进行，最后在阶段 5 汇合。

## 4. 各阶段技术方案

### 4.1 音频提取

- **工具**：FFmpeg（`ffmpeg -i input.mp4 -vn -acodec pcm_s16le -ar 16000 -ac 1 output.wav`）
- **输出**：16kHz 单声道 WAV 文件（ASR 模型标准输入格式）
- **成本**：免费

### 4.2 语音转文字 (ASR)

#### 选型决定

| 优先级 | 方案 | 模型 | 质量 | 1h 视频耗时 | 运行位置 |
|--------|------|------|------|-----------|---------|
| **首选** | faster-whisper | large-v3-turbo | WER ~14% (多语言) | ~90 秒 | RTX4070Ti (CUDA) |
| 备选 | Qwen3-ASR-1.7B | Q4 量化 | WER ~5-10% (中英) | ~60 秒 | RTX4070Ti (CUDA) |
| Mac 备选 | mlx-whisper | large-v3-turbo | 同 faster-whisper turbo | ~2-3 分钟 | Mac M4 (MLX) |
| Mac 备选 | OminiX-MLX | Qwen3-ASR-1.7B Q8 | 同 Qwen3-ASR | ~3 分钟 | Mac M4 (MLX) |

#### 关键约束

1. **必须使用 `task="transcribe"` 模式**（不是 translate）。turbo 模型未训练翻译任务，翻译模式会导致幻觉循环。
2. faster-whisper turbo 在中英混合场景下，语言切换点可能有信息丢失。这是 Whisper 系列的已知局限。
3. Qwen3-ASR 对中英混合场景支持更好（52 种语言 + 22 种中文方言），但需要 Q4 量化才能在 12GB VRAM 上运行。

#### 不选择的方案及原因

| 方案 | 不选原因 |
|------|---------|
| FunASR | 纯中文场景最强，但中英混合无基准数据，热词限制 10 字符无法覆盖技术术语 |
| faster-whisper small | 准确度不如 turbo（WER 16.1% vs 14.0%），而显存不是瓶颈，没有理由用更差的模型 |
| faster-whisper large-v3 | turbo 准确度仅差 0.1-0.4%，但速度慢 8 倍，不值得 |

### 4.3 关键帧提取

- **工具**：FFmpeg（`ffmpeg -skip_frame nokey -i input.mp4 -vsync vfr -frame_pts true keyframes/key_%06d.png`）
- **输出**：PNG 关键帧图片序列（文件名包含帧时间戳）
- **提取策略**：使用关键帧（I-frame）提取，关键帧是视频中存储的完整图像，质量高于中间帧
- **成本**：免费

#### 关键帧筛选（去重）

提取的原始关键帧可能存在大量重复（静态画面）。需要去重策略：

1. 使用感知哈希（pHash）计算相邻帧的相似度
2. 相似度超过阈值（如 95%）的帧只保留第一帧
3. 最终保留的关键帧数量预期：1 小时视频约 50-200 帧（取决于内容变化频率）

### 4.4 关键帧视觉分析

#### 策略决定：全部使用收费多模态模型

**不使用 OCR 方案**，全部关键帧使用智谱视觉模型分析。

理由：
- 成本极低（GLM-4.6V-FlashX 约 ¥0.003/视频），不值得为了省钱引入 OCR 管道的复杂度
- 多模态模型能理解语义（代码结构、函数调用关系、图表含义），OCR 只能提取文字
- 多模态模型对 IDA Pro 等复杂界面效果远优于 OCR
- 少一套技术依赖，维护成本更低

#### 模型选择

| 模型 | 输入价 (元/百万tokens) | 输出价 (元/百万tokens) | 60 帧估算成本 | 推荐场景 |
|------|----------------------|----------------------|-------------|---------|
| **GLM-4.6V-FlashX** | 0.03 | 0.03 | ~¥0.003 | **默认选择**，极致性价比 |
| GLM-4.6V-Flash | 免费 | 免费 | ¥0 | 免费试用 / 效果验证 |
| GLM-4.6V | 0.2 | 0.4 | ~¥0.02 | FlashX 不满意时升级 |
| GLM-5V-Turbo | 1.2 | 22 | ~¥0.12 | 最强理解力，复杂帧兜底 |

> 价格基于输入长度 [0,32) 区间的限时免费价。实际价格取决于图片分辨率对应的 token 消耗。
> 上述价格是 2026 年 4 月的限时免费价格，未来可能调整，但即使按标准价也极便宜。

#### 提示词设计要点

对每个关键帧，视觉模型需要输出：
1. 画面类型分类（IDE / 终端 / PPT / IDA Pro / 其他）
2. 画面中的文字内容（代码则完整提取，含缩进）
3. 画面语义描述（如：正在讲解某个函数的逻辑、展示调用关系图）
4. 对应时间戳（从文件名中获取）

### 4.5 AI 综合总结

- **模型**：GLM-4.7 或 GLM-5.1（使用智谱 Max Coding Plan 配额，不额外花钱）
- **输入**：
  - ASR 转录文本（带时间戳，按时间轴排列）
  - 视觉分析结果（带时间戳，按时间轴排列）
  - 视频文件名（作为上下文参考）
- **输出**：结构化 Markdown 文档

#### 总结 Prompt 设计要点

Prompt 应指导 AI 生成以下结构的 Markdown：

```markdown
# [视频标题/文件名] 内容总结

## 概述
[2-3 句话的视频核心内容概括]

## 详细内容

### [主题/章节 1]
[讲解内容总结，包含关键概念解释]

**相关代码示例：**
```[语言]
[从视觉分析中提取的代码，或 AI 根据讲解内容生成的示例代码]
```

### [主题/章节 2]
...

## 关键要点
- [要点 1]
- [要点 2]
- ...

## 截图引用
![描述](./screenshots/key_001234.png)
...
```

### 5.2 全局 YAML 配置文件

配置文件路径：`config/config.yml`

所有可能切换的模型、策略、参数都必须通过配置文件控制。**不允许在代码中硬编码模型名称或硬件相关的参数。**

```yaml
# ========== 运行环境配置 ==========
# env: auto          # auto = 自动检测，可强制指定 mac / windows_gpu
# auto_detect: true  # 是否自动检测硬件

# ========== ASR 语音转文字配置 ==========
asr:
  # 可选引擎: faster_whisper / qwen3_asr / mlx_whisper
  engine: faster_whisper

  # 各引擎的配置，运行时根据 engine 字段选择对应的配置节
  engines:
    faster_whisper:
      # 可选模型: large-v3-turbo / small / medium / large-v3
      model: large-v3-turbo
      # 可选计算精度: float16 / int8
      compute_type: float16
      # 可选设备: cuda / cpu / mps（auto = 自动根据环境选择）
      device: auto
      # 中英混合必须用 transcribe，禁止用 translate
      task: transcribe
      # 语言设置，None 表示自动检测
      language: null
      # 不同环境下的覆盖配置
      env_overrides:
        mac:
          # Mac 上优先用 MLX 后端，而不是 faster-whisper
          # 如果用 faster-whisper 则只能跑 CPU，速度较慢
          device: cpu
          compute_type: int8
        windows_gpu:
          device: cuda
          compute_type: float16

    qwen3_asr:
      model: Qwen/Qwen3-ASR-1.7B
      # 可选量化: fp16 / q8 / q4
      quantization: q4
      device: auto
      # vLLM 后端（仅 CUDA 可用）
      backend: vllm
      gpu_memory_utilization: 0.8
      env_overrides:
        mac:
          # Mac 上需用 OminiX-MLX，不支持 vLLM
          backend: omlx
          quantization: q8
        windows_gpu:
          backend: vllm
          quantization: q4

    mlx_whisper:
      # Mac 专用，使用 MLX 框架加速
      model: large-v3-turbo
      # MLX 自动使用 GPU (Metal)
      language: null
      task: transcribe
      env_overrides:
        mac:
          model: large-v3-turbo
        windows_gpu:
          # Windows 不支持 MLX，不应使用此引擎
          enabled: false

# ========== 关键帧提取配置 ==========
keyframe:
  # 提取方式: keyframe (I-frame) / interval (定时)
  method: keyframe
  # 输出图片格式
  format: png
  # 去重阈值 (0-1)，感知哈希相似度超过此值的相邻帧视为重复
  dedup_threshold: 0.95
  # 最大提取帧数（防止超长视频产生过多帧）
  max_frames: 500

# ========== 视觉分析配置 ==========
visual:
  # 可选模型: glm-4.6v-flashx / glm-4.6v-flash / glm-4.6v / glm-5v-turbo / glm-4.5v
  model: glm-4.6v-flashx
  # 智谱 API Key（也可通过环境变量 ZHIPU_API_KEY 设置）
  api_key: null
  # 单帧分析的最大 token 输出
  max_tokens: 2000
  # 并发请求数（控制 API 调用速率）
  concurrency: 5
  # 请求超时（秒）
  timeout: 30
  # 失败重试次数
  retry: 3

# ========== AI 综合总结配置 ==========
summary:
  # 可选模型: glm-5.1 / glm-5 / glm-5-turbo / glm-4.7 / glm-4.6 / glm-4.5-air
  model: glm-4.7
  # 智谱 API Key（同 visual，也可通过环境变量设置）
  api_key: null
  # 最大输出 token 数
  max_tokens: 8000
  # 请求超时（秒）
  timeout: 120
  # 失败重试次数
  retry: 3
  # 总结 Prompt 模板文件路径（相对于 config 目录）
  # null 表示使用内置默认 Prompt
  prompt_template: null

# ========== 输出配置 ==========
output:
  # 输出根目录（null = 当前目录下的 output/）
  dir: null
  # 是否在 Markdown 中嵌入截图引用
  include_screenshots: true
  # 截图引用路径（相对于 Markdown 文件的路径）
  screenshot_rel_path: ./keyframes/
  # 是否保留中间文件（audio.wav, transcript.json 等）
  keep_intermediate: true

# ========== 日志配置 ==========
logging:
  level: INFO
  # 日志格式
  format: "%(asctime)s [%(levelname)s] %(name)s: %(message)s"
```

#### 配置加载优先级

1. 默认值（代码中定义）
2. `config/config.yml` 文件
3. 环境变量覆盖（如 `ZHIPU_API_KEY`）
4. CLI 参数覆盖（如 `--model glm-5.1`）

后加载的优先级更高，覆盖前面的配置。

#### 环境覆盖机制

配置中的 `env_overrides` 字段允许针对不同运行环境设置不同的参数。加载逻辑：

```python
# 伪代码
config = load_yaml("config/config.yml")
current_env = context.env  # "mac" 或 "windows_gpu"

# 对每个包含 env_overrides 的配置节，合并环境特定配置
for section in walk_config(config):
    if "env_overrides" in section:
        override = section["env_overrides"].get(current_env, {})
        section.update(override)
        del section["env_overrides"]  # 合并后移除
```

### 5.3 全局上下文对象

程序启动时创建全局上下文对象，所有模块通过此对象获取运行时环境信息。

```python
@dataclass
class RuntimeContext:
    """全局运行时上下文，程序启动时初始化一次"""

    # 运行环境
    env: str                    # "mac" 或 "windows_gpu"
    os_name: str                # "darwin" 或 "windows"
    has_cuda: bool              # 是否有可用的 CUDA GPU
    has_mps: bool               # 是否有可用的 Apple MPS
    gpu_name: str | None        # GPU 名称，如 "NVIDIA GeForce RTX 4070 Ti"
    gpu_vram_mb: int | None     # GPU 显存大小 (MB)
    cpu_count: int              # CPU 核心数
    total_ram_gb: float         # 总内存大小 (GB)

    # 运行配置（合并后）
    config: dict                # 加载并合并环境覆盖后的完整配置

    # 路径
    output_dir: Path            # 输出目录
    temp_dir: Path              # 临时文件目录
```

#### 上下文初始化流程

```
main.py 启动
  │
  ├── 1. 加载 YAML 配置
  ├── 2. 检测运行环境 (env_detector.py)
  │     ├── 检测操作系统 (platform.system())
  │     ├── 检测 CUDA (torch.cuda.is_available())
  │     ├── 检测 GPU 信息 (torch.cuda.get_device_name / get_device_properties)
  │     ├── 检测 Apple MPS (torch.backends.mps.is_available())
  │     ├── 检测 CPU / 内存 (psutil)
  │     └── 判定环境类型: mac / windows_gpu
  ├── 3. 合并环境覆盖配置
  ├── 4. 创建 RuntimeContext 全局实例
  └── 5. 传递 context 给管道各阶段
```

### 5.4 环境检测逻辑

```python
def detect_environment() -> str:
    """检测当前运行环境，返回 "mac" 或 "windows_gpu" """

    system = platform.system()

    if system == "Darwin":
        # macOS (Apple Silicon)
        return "mac"

    if system == "Windows" or system == "Linux":
        if torch.cuda.is_available():
            gpu_name = torch.cuda.get_device_name(0)
            vram = torch.cuda.get_device_properties(0).total_mem
            # 检测到 NVIDIA GPU，使用 GPU 模式
            return "windows_gpu"

    # 无 GPU 或不支持的平台
    return "mac"  # 回退到 CPU 模式
```

#### 基于环境的自动路由

| 环境检测结果 | ASR 引擎自动选择 | 关键帧分析 | 其他 |
|------------|-----------------|-----------|------|
| Mac (无 CUDA) | mlx-whisper 或 faster-whisper (CPU) | 智谱 API | ASR 速度较慢但可用 |
| Windows + RTX4070Ti | faster-whisper (CUDA) | 智谱 API | ASR 速度快，首选 |

**ASR 引擎自动选择逻辑**：

```python
def resolve_asr_engine(context: RuntimeContext) -> str:
    """根据环境自动选择最优 ASR 引擎"""

    # 用户在配置中明确指定了引擎，优先使用
    if context.config["asr"]["engine"] != "auto":
        return context.config["asr"]["engine"]

    # 自动选择
    if context.has_cuda:
        return "faster_whisper"   # CUDA 下 faster-whisper 速度最快
    elif context.os_name == "darwin":
        return "mlx_whisper"      # Mac 上 MLX 框架最快
    else:
        return "faster_whisper"   # 回退到 CPU
```

## 8. 错误处理策略

| 场景 | 处理方式 |
|------|---------|
| FFmpeg 未安装 | 启动时检测，给出安装提示后退出 |
| CUDA 不可用 | 自动回退到 CPU 模式，打印警告 |
| 智谱 API 调用失败 | 重试 3 次（指数退避），仍然失败则跳过该帧并记录错误 |
| ASR 转写部分失败 | 保留已转写部分，在总结 Prompt 中标注缺失段落 |
| 视频文件无法读取 | 报错退出，提示文件格式支持列表 |
| 显存不足 (OOM) | 自动降低 batch_size，若仍失败则回退到 CPU |
| 配置文件不存在 | 使用内置默认配置，打印警告 |
